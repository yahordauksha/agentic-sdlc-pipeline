# Timeline

Narrative log of when and why this pipeline changed shape. History, not active work — add a short entry when something structural changes.

## 2026-07-07 — Vault created; first two templates ported

Split out of the `product/argus` code repo after noticing the pipeline's core patterns (supervisor dispatch, adversarial audit, dependency verification before parallelizing) were genuinely project-agnostic, while only a thin layer (domain invariants, issue-tracker specifics) was Argus-specific.

Same day, the Argus `implementer` **Agent** was converted to an `/implement` **Skill** — a structural fix, not a rename. The agent had a documented, twice-recurring failure (bailed real issues #337 and #78) because a subagent has no `Agent`/`Task` tool of its own and silently couldn't dispatch its specialists when spawned that way. A preflight-check workaround (STEP -1) had been drafted but was still unmerged on the third recurrence. Converting it to a Skill removes the failure mode structurally — see [[Principles#Skill vs. Agent: a structural decision, not a style preference]].

Ported to this vault: [[Templates/Skills/implement]] (the supervisor) and [[Templates/Agents/edge-case-auditor]] (the richest single pattern — adversarial, intent-first verification). Still to port: `core-implementer`, `test-writer`, `doc-keeper`, `iac-specialist`, `triage`, `queue-scout`, `spec`.

## 2026-07-07 (later) — Remaining specialists ported; two real incidents added to Principles

Same day, back on the Argus repo: a proposed "reconciliation guardian" agent (from a usage-insights report) turned out to mostly duplicate two checks that already existed — `/implement`'s dependency verification and `queue-scout`'s bulk version of the same. Rather than stand up a redundant agent, the actual new idea (checking that acceptance criteria are *semantically* satisfiable, not just that a named field exists) got folded into both as a new sharpened step, citing a real precedent (`#349` — a notification filter that assumed a field would be set, but the field was only ever populated by an action attached to the same notification, so it was always empty when the filter needed it). See [[Principles#Ground truth over expectation]].

Giving `/spec` a matching capability — drafting an ADR when it resolves a genuine drift-caused bail, instead of just quietly correcting the spec text — required adding filesystem write and git commit/push to a skill that had never had either. The first attempt bundled that permission change into the same edit as the instructional content and was correctly blocked by an automated permission gate: "sharpen the checks" didn't specifically authorize new git/write access. Asking directly, separately, got an explicit yes. This is now its own principle — see [[Principles#Capability grants are their own decision]].

Ported the remaining five agents ([[Templates/Agents/core-implementer]], [[Templates/Agents/test-writer]], [[Templates/Agents/doc-keeper]], [[Templates/Agents/iac-specialist]], [[Templates/Agents/triage]]) and both remaining skills ([[Templates/Skills/queue-scout]], [[Templates/Skills/spec]]), all carrying the sharpened checks and the two-step-confirmation ADR capability from the start rather than needing a later pass. All seven specialists plus both skills are now in the vault.

## 2026-07-13 — Vault identity clarified; drift discovered and closed; two-way sync automated

Prompted by a direct question — "what even is this repo, and why do I have to manually tell you to update it" — an audit against the live Argus repo found this vault had already drifted: four agents (`ecosystem-diagnostician`, `ops-check`, `vault-auditor`, `vault-sync`) and two skills (`cleanup`, `pipeline-review`) existed in Argus's `.claude/` with no `Templates/` counterpart at all. Ported and generalized all six (see each file's own history), following this vault's existing `[CUSTOMIZE]`-marker convention.

Built the mechanism that should have caught that drift automatically: `ecosystem-sync` (see [[Templates/Agents/ecosystem-sync]]), modeled directly on `vault-sync`'s own automatic/narrow shape but pointed at this vault's `Templates/` instead of a product-strategy vault. Installed a concrete instance in Argus's `.claude/agents/`, and wired a new STEP 7d into Argus's `implement.md` so every PR touching `.claude/agents/*`/`.claude/commands/*` dispatches it automatically — the old self-review step, previously STEP 7d, is now STEP 7e. This closes the "I have to manually tell you to update the vault" gap going forward; see [[Principles#A sync mechanism needs both an automatic-narrow check and a periodic-broad sweep]] for why this pairing (automatic-narrow now, periodic-broad still an open question) is the right shape rather than either half alone.

Built `/onboard` (`.claude/commands/onboard.md`), this vault's own skill for the other direction: given a target project's path, it reads the project for real evidence, proposes which templates apply, fills in `[CUSTOMIZE]` markers from what it actually found (never a guess), and writes the result after explicit confirmation — automating [[Adoption Checklist]] steps 1-4 without skipping the judgment calls those steps exist to make.

Rewrote `Home.md` and `CLAUDE.md` to state the vault's identity plainly: the canonical reference for every agent/skill in this pipeline, not a static extraction from Argus. Corrected a stale line in both — "templates should only encode patterns proven across more than one project" didn't match how any template here actually got in (every one was ported after exactly one project's use); the real bar, now stated directly, is "structurally reusable," not "already proven twice."

## 2026-07-14 — GitHub interaction layer consolidated; a general shared-shape principle named

A prioritization pass over Ideas.md flagged the unified-GitHub-interaction-layer idea as the highest-leverage item in the backlog — pure de-duplication, no new design. Auditing the actual files to consolidate it turned up more than style drift: two comment shapes (`implement`'s BAIL comment, the recurring-pattern comment `pipeline-review`/`ecosystem-diagnostician` both assume exists) and the ecosystem-fix tracking issue's title convention were each referenced by name from three separate files and defined as literal text in none of them — every file assumed one of the others had written it down.

Built [[Templates/GitHub/labels]], [[Templates/GitHub/issue-templates]], [[Templates/GitHub/comment-templates]] as the canonical definitions, grounded in real prior art rather than invented from scratch: comment syntax borrows [Conventional Comments](https://conventionalcomments.org/)' `label (decorations): subject` shape, the label taxonomy's colon-namespacing and the issue templates' essential-fields-only discipline both follow GitHub's own documented conventions. Refactored `triage`, `spec`, `implement`, `ops-check`, `pipeline-review`, and `ecosystem-diagnostician` to reference these instead of each re-describing (or, in the two undefined cases, silently assuming) its own copy — also fixing a live naming inconsistency along the way (`ops-check` spelled the safety-surface label `safe-surface:yes|no`; every other file just said "the safety-surface label" without ever giving the concrete slug).

Generalized the pattern into a new principle — see [[Principles#Shared shapes need one definition, not one description per reader]] — since "if it's referenced by more than one file, it needs one written definition" is a real, recurring trigger this vault will keep hitting as more templates get added, not a one-off GitHub-specific fix. Deliberately scoped narrower than "template everything": the principle explicitly excludes judgment-heavy free text (`edge-case-auditor` findings, `ecosystem-diagnostician`'s reasoning) from being scripted into fill-in-the-blank shape.

Same day: four smaller items closed out from Ideas.md. A new principle — [[Principles#Ground new additions in something citable]] — now states plainly what this session had already been doing informally: a proposed `security-reviewer` agent turned out to fully duplicate Claude Code's own built-in `security-review` skill, so that idea closed as "already covered" rather than built. `spec.md` and [[Templates/GitHub/issue-templates]] now require naming a rejected alternative when one was genuinely considered, for both the "decisions made during spec" field and any drift-driven ADR. And `implement.md`/`ops-check.md` got the narrow slice of notification routing actually worth building right now: a `PushNotification` on a `safe-surface:yes` BAIL or a new `severity:critical` finding specifically, nothing wider — the full interrupt/digest/log routing layer from the original idea would have formalized tiers that already exist informally (`needs:decision` as the digest, periodic-review skills as the log), so only the genuinely missing interrupt tier got built.

Same day: this repo got its first CI. Two different things, not one — this vault's own `scripts/lint_vault.py` + `.github/workflows/vault-lint.yml` (structural lint: every wiki-link resolves, every Templates/Agents+Skills file has required frontmatter — dogfooded clean against the whole vault, and it already caught two real pre-existing bugs: a broken `[[Principles#...]]` link in `vault-auditor.md` that should have referenced the Operator's global CLAUDE.md rule as plain text instead, and a stale premise in this same GitHub-interaction-layer entry above about `pipeline-review` assuming six named workflows — re-checking the actual file found it only ever said "[CUSTOMIZE] adapt to your CI provider," never named a single one) — versus [[Templates/CI/README]] (`ci.yml`, an export for adopting projects). The export was scoped down from the original six-workflow idea to just the one thing actually grounded in an existing template (`implement.md` STEP 7c's self-review step) — building `deploy`/`auto-merge`/`auto-rebase`/`pr-review`/`issue-triage` templates now would have been seeding speculative capability against no real dependency, the same pattern this vault already argues against for cross-project telemetry in [[Principles]].

## 2026-07-14 (later) — Deep research into safe agent-editing practices; `prompt-implementer` built; two guardrail principles added

A design question — should this pipeline gain a skill that shapes a raw idea all the way to implementation, including ideas about the pipeline's own tooling, not just product code — surfaced a real gap: `implement.md`'s specialist trio (`core-implementer`/`test-writer`/`edge-case-auditor`) is entirely code-shaped and has no answer for "the implementation is a change to an agent's own prompt." A deep-research pass (5 angles, ~100 sources fetched, 3-vote adversarial verification per claim) went looking for established practice before designing one from scratch.

Worth recording honestly: the research workflow's own final synthesis step failed twice — once returning a literal placeholder ("test") instead of a report, once hard-erroring after 5 failed structured-output attempts — despite the underlying search/fetch/verify work being real and substantial (over 100 genuinely verified claims across both runs). Diagnosed by reading the raw run journal directly rather than trusting the tool's own summary, per the standing rule that a workflow's cached result can itself be empty or broken. Rather than pay for a third full automated run, the actual synthesis was done by hand from the verified raw data — a good instance of the research's own conclusion: the flashy automated eval/reporting tooling's specific performance claims don't always hold up, plain manual verification against real evidence sometimes just wins.

Real findings that changed this vault's own design, not just `prompt-implementer`'s:
- A documented incident (Meta's OpenClaw agent, ordinary context-window compression silently dropping a 10-token safety constraint out of 50,000 tokens of conversation, leading to 200+ emails deleted) and a governance-research finding (durable constraints on a self-modifying agent must be institutional — an audit trail, a review board — not architectural, because a capable-enough agent can modify a code-based constraint the same way it modifies anything else) — together became [[Principles#Guardrails need an anchor outside the agent's own context, not just good prompting]], naming why several of this vault's existing patterns (CI, `spec.md`'s two-step ADR confirmation, the independent-audit pattern) are shaped the way they already are.
- A real red-teaming study ("Agents of Chaos," arXiv 2602.20021) documenting agents self-reporting task completion while system state contradicted it, as one of eleven named failure modes — sharpened [[Principles#Adversarial verification: assume both are wrong]] with real-world evidence beyond the original Argus incident, generalizing it explicitly to any agent's self-report, not just code review.
- A concrete, citable operational test for "how much scrutiny does this change need" — reversibility, and whether state/data/an external workflow actually changes, the same test ITIL's decades-old change-management practice already tiers on — sharpened [[Principles#Safety surfaces need a bright line, not a judgment call]] and became the reversibility-tiering (LOW/MEDIUM/HIGH) at the center of [[Templates/Agents/prompt-implementer]]'s design.

Built [[Templates/Agents/prompt-implementer]]: tiers every edit by reversibility, never writes a HIGH-tier change without returning `NEEDS-CONFIRMATION` for explicit Operator sign-off first (mirroring `spec.md`'s own ADR pattern), never grades its own medium/high-tier work (explicitly deferring the independent audit to whatever skill dispatches it, reusing `edge-case-auditor`'s exact discipline rather than inventing a parallel one), and runs a manual scenario trace instead of taking a dependency on an automated agentic-eval framework — directly because this research found those frameworks' specific performance claims didn't survive adversarial verification. Added a light cross-reference from `iac-specialist.md`'s existing plan-only discipline, since it had already independently arrived at the same reversibility test.

Built [[Templates/Skills/shape]] the same day, closing the loop this whole design thread started from: the operator-facing front door for one idea at a time, grounded in established practice rather than invented — ICE (Sean Ellis, GrowthHackers) by default, RICE (Intercom) only when Reach is a genuinely knowable number; a real Klein-protocol pre-mortem (2007 HBR, the "assume it already failed" tense shift that measurably improves risk-finding over "what could go wrong"); pre-registered kill criteria, since portfolio-management practice shows a documented trigger gets acted on far more often than an undocumented judgment call once real work is sunk into an idea; Amazon's PR/FAQ available but deliberately gated to major bets only, matching how expensive Amazon's own process actually treats it. Resolves the idea-red-teaming entry that used to live in Ideas.md (since retired — see below). Routes a surviving idea to `/implement` for product code or to [[Templates/Agents/prompt-implementer]] plus a separately-dispatched audit for this pipeline's own tooling — never silently auto-chaining into either, per [[Principles#Consent boundary: analysis is not authorization to act]].

## 2026-07-14 (later still) — prompt-implementer's independent auditor built; `/shape` installed live; `Ideas.md` retired to GitHub issues

Asked directly whether `/shape` would actually be usable — it wasn't yet, since everything in `Templates/` is a generalized reference, not something callable. That surfaced the real next step: install customized, live copies rather than leave the design sitting in the vault as documentation. Also decided, correcting an earlier assumption: intake should be real GitHub issues, and `Ideas.md`'s own backlog should migrate there too rather than staying a parallel, uncustomized-by-a-tracker markdown dump — every idea gets the same paper trail regardless of where it originated, the same principle `/shape` already applies per-idea, now applied to this vault's whole backlog at once.

Built [[Templates/Agents/prompt-auditor]] first, since `prompt-implementer`'s own design already required an independent auditor that didn't exist yet — grounded in a quick supplementary research pass (LLM-as-judge self-preference bias is real and measurable; the standard mitigation, cross-model-family majority voting, isn't available in an environment with only one model family, so the auditor leans on independent-reasoning-path derivation instead, plus running twice for HIGH-tier edits since self-consistency across repeated runs is itself a citable technique; checklist-driven review measurably catches more real defects than unstructured review). `/shape` now dispatches `prompt-auditor` once for LOW/MEDIUM tier, twice independently for HIGH tier, surfacing disagreement between the two passes rather than averaging it away.

Installed live, customized copies — `.claude/commands/shape.md`, `.claude/agents/prompt-implementer.md`, `.claude/agents/prompt-auditor.md` — wired to `gh issue`, not a placeholder issue-tracker CLI. Confirmed via `gh auth status`/`gh repo view` this repo already has a working GitHub remote before customizing anything against it.

Migrated `Ideas.md`'s still-open entries into 35 GitHub issues (already-resolved entries were left out — their record already lives in this file, not duplicated as issues). Retired `Ideas.md` to a short redirect notice rather than deleting it outright, per this vault's own "never delete, mark how it was resolved" discipline (see `vault-auditor.md`'s HARD RULES) — the several existing `Ideas.md` cross-references across this vault kept resolving; the two that pointed at a specific now-gone heading were rewritten rather than left dangling.

## 2026-07-15 — Project-lifecycle territory consolidated; the real reason `/onboard` v1 died, finally recorded

A conversation that started narrow — `/spec` has no mode for novel/complex architectural design (#53) — broadened once the Operator pointed out the deeper gap: nothing in this vault covers the moment *before* a project exists at all, upstream of `/shape` and `/spec` both. Grounded the redesign in real evidence rather than invented shape: compared `argus/docs` against `map_of_telegram/docs` directly (both independently converged on the same Home/Ambiguities/Research-Queue/Timeline/Ideas-&-Experiments core), checked `argus/product/argus` (the actual origin of this vault's Templates) for what a code-repo scaffold needs, and discovered Argus is actually four independent git repos under one umbrella (docs/landing/product/review), not two. Also reviewed a separate agent-ecosystem project on disk (`superflow`) for prior art — its Phase 0 greenfield/onboarding flow, expert-panel design-brainstorming pattern, and per-stack scaffold templates all turned out to be concrete, reusable precedent, folded into the relevant issues as grounding.

Filed #55/#56/#57 as "new" issues mid-conversation without first searching the existing backlog — a real mistake, caught by the Operator, not self-caught. They substantially duplicated #2, #5, #6, #7, several already migrated from the old `Ideas.md` backlog. Closed all six (#2, #5, #55, #56, #57, #59) as superseded, consolidated into two canonical issues: #62 (project-genesis + GitHub-provisioning) and #63 (project-retrofit), architected as one skill with internal state-detection rather than several separate skills — resolving #5's long-open "one orchestrator or three skills" question.

Also got, for the first time, the Operator's actual account of why `/onboard` v1 was pulled same-day (this file's 2026-07-14 entry only ever recorded "didn't sit right" — the reason itself was never written down): it was built without being asked, and it installed every template into every project regardless of relevance. Both are now hard requirements on #62/#63 rather than folklore — see [[Principles#Template distribution must be relevance-scored, never install-everything]].

Separately: #6 (adopter registry) and #31 (template semver) shipped this same day via a parallel session, both now real precedent grounding #62/#63/#7/#60 rather than speculative design. #6's actual shipped schema (identity per-repo URL, grouped by project name only at registration) settled the multi-repo-per-project question: one repo per invocation, not an umbrella abstraction. #37 (investigate whether Claude Code's plugin/marketplace system already solves push-updates/versioning/registry) was found to have been shaped with an explicit disposition that its verdict should post to #6/#7/#31 before they were built — that never happened; #6/#31 shipped anyway. Re-shaped from scratch and resolved later this same day: plugin skills turned out to already be push-update-solved (local cache + auto-sync), and plugin agents support real per-install customization via `userConfig` — but only for short scalar fields, not this vault's prose-heavy `[CUSTOMIZE]` placeholders — so adopting it now isn't worth the cost (a template rewrite plus new marketplace infrastructure this vault doesn't have) given current adopter count; #62/#63 proceed with the existing copy-based model. See [[Principles#Claude Code's own plugin system already solves push-updates for skills — but not, right now, for this vault's agent customization]].
