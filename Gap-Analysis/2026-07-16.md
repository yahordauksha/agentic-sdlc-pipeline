## Run-quality notes (read before trusting the finding below)

> [!warning] This run was stopped mid-execution by the Operator, and most of it never happened
> Following this skill's Step 2 and Step 4 literally, I queued the categories.md "what's new" pass plus one `deep-research` dispatch per category — 4 were actually launched (the "what's new" pass, plus category comparisons for multi-agent orchestration frameworks, agentic coding prior art, and durable execution) before the Operator stopped all of them. Each `deep-research` dispatch fans out into a 5-phase pipeline (5 parallel search agents → fetch up to 15 sources → 3-vote adversarial verification per extracted claim) — 4 dispatches in parallel meant on the order of hundreds of subagent calls for what the Operator had asked to be a lightweight "save some findings" task. That mismatch between what Step 4 says ("one dispatch per category") and what one dispatch actually costs was never surfaced before launching — a real process failure, not a tooling glitch. The 3 remaining planned dispatches (memory management, guardrails/safety, and a deeper BMAD-METHOD comparison) were never launched at all.

> [!note] The stopped dispatches' partial output was deliberately not mined for this report
> Each stopped workflow's journal contains dozens of real, completed sub-results — individual search results and per-claim adversarial-verification votes. But none of the 4 reached their synthesis/dedup/ranking stage before being stopped. A single verification vote on one extracted claim, read in isolation without the workflow's own multi-vote consensus and dedup pass, is not the same thing as a vetted finding — presenting fragments as if they were synthesized findings would violate this skill's own kill-criterion-4 discipline (every finding must be independently verified, not just cited). So: zero findings are reported from those 4 stopped dispatches, not a diluted or reconstructed set.

> [!note] Categories 1 and 7 carried forward, not re-researched
> "Agentic design patterns" (confirmed no gap, 2026-07-15) and "Interoperability/Standardization Protocols" (confirmed no current relevance, 2026-07-15) were not re-dispatched this run — both were resolved with real citations last run, and nothing in this vault's design has changed since that would reopen either question. The candidate fix (a global CLAUDE.md rule requiring fan-out scale to be disclosed before invoking `deep-research` or similar heavy dispatchers, especially more than one in parallel) is still an open proposal awaiting the Operator's decision — tracked in this session, not written into this report as a vault finding since it's about tool-invocation discipline, not this vault's Templates/Agents/Templates/Skills design.

## Category 8 — Opinionated multi-agent SDLC pipeline frameworks (new this run)

> [!tip] Confirmed — no gap found; if anything, a candidate differentiator worth naming plainly
> Researched via direct web search (not `deep-research`'s adversarial-verification pipeline — see the citation-confidence caveat above). BMAD-METHOD (bmad-code-org/BMAD-METHOD, 43K+ stars) is the closest external match found to date, closer than `superflow`: role-based agents (PM/Architect/Dev/QA) producing persistent, versioned artifacts (PRDs, architecture docs, story files) with quality gates between phases, and an expansion-pack system for porting domain knowledge into new projects. GitHub Spec Kit (github/spec-kit) runs a comparable `specify → plan → tasks → implement` phase structure with human checkpoints, structurally close to this vault's `/shape` → `/spec` → `/implement` flow. Agent OS (buildermethods/agent-os) injects durable codebase standards into agent context. Specky (paulasilvatech/specky) runs 13 agents behind a deterministic 10-phase pipeline. MetaGPT (FoundationAgents/MetaGPT) is the original of this pattern but is a standalone Python framework, not built for a coding agent's file-based agent/skill system.
>
> What none of these appear to have, based on what surfaced: this vault's **bidirectional** sync — [[Templates/Agents/ecosystem-sync]] auto-ports genuinely-reusable `.claude/agents`/`.claude/commands` changes from an adopting project *back* into this shared template vault on every relevant PR, no manual step. BMAD's expansion packs are shared *outward* (upstream → project); no evidence surfaced of an equivalent automated *inward* path (project → upstream). This is a candidate differentiator worth being aware of if this vault is ever positioned publicly — not a gap to close, and not confirmed with the same rigor as a normal `deep-research`-verified finding, so treat it as a lead rather than a settled fact.

Sources: https://github.com/bmad-code-org/bmad-method, https://github.com/github/spec-kit, https://github.com/buildermethods/agent-os, https://github.com/paulasilvatech/specky, https://github.com/FoundationAgents/MetaGPT
