## Category 1 — Agentic design patterns (Anthropic's "Building Effective Agents")

> [!tip] Already covered — no gap
> Confirmed against the primary source directly (https://www.anthropic.com/engineering/building-effective-agents, verified via 6 independent votes, all consistent): orchestrator-workers is "a central LLM [that] dynamically breaks down tasks, delegates them to worker LLMs, and synthesizes their results," distinguished from parallelization because "subtasks aren't pre-defined, but determined by the orchestrator based on the specific input." Evaluator-optimizer is "one LLM call generates a response while another provides evaluation and feedback in a loop."

## Category 2/4 — Durable/resumable execution (big-review finding)

> [!warning] `implement.md`'s BAIL procedure fully discards work; it does not resume from a checkpoint
> [[Templates/Skills/implement]] STEP 8 (BAIL procedure), step 1: **"Discard all local work (remove the worktree and branch)."** On any specialist BAIL, the entire worktree is torn down and the issue goes back to a "needs shaping" state for a human-reviewed respec — none of the already-completed specialist work (e.g. a successful core-implementer pass, if test-writer is what bailed) is preserved or resumed.

Compare against LangGraph's actual mechanism (confirmed via LangGraph's own docs, https://docs.langchain.com/oss/python/langgraph/persistence and https://reference.langchain.com/python/langgraph/checkpoints): checkpoints are keyed by `thread_id`, and "LangGraph recovers from partial workflow failures via 'pending writes': writes from successful nodes are preserved when other nodes fail, allowing resumption without re-running completed work." Separately, OpenHands' actual persistence mechanism — contrary to a popular secondary-source claim that it's "event-stream replay instead of a state file" (that specific claim was directly refuted twice, independently, against OpenHands' own official docs) — is a **hybrid**: a `base_state.json` snapshot loaded first, then an event log replayed on top of it, explicitly "without replaying the entire event stream from scratch" (https://docs.openhands.dev/sdk/guides/convo-persistence).

This is a genuine, citable gap: two real external systems (LangGraph, OpenHands) preserve completed work across an interruption and resume from where they left off; this vault's own pipeline treats any specialist failure as a full restart requiring a fresh human-reviewed spec. This is issue #24's own original callout, though its framing ("hand-rolled worktree-resume-on-bail logic") turned out to be inaccurate against the current file — `implement.md` doesn't attempt to resume-on-bail at all; it discards unconditionally. Per [[Principles#Ground truth over expectation]], correcting the premise rather than repeating it.

**Not filed as an issue** — this is a genuine design tradeoff (worktree-discard-and-respec is arguably a deliberate simplicity choice, not an oversight — see [[Principles#Incremental discipline scales trust]] and the general "simplicity first" bias this vault otherwise follows) spanning the whole BAIL architecture, not a single well-scoped fix. Worth the Operator's judgment on whether the added complexity of a checkpoint-preserving BAIL is worth it here, not a one-line correction.
