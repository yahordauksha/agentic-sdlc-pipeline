# Principles

The design rules behind this agent ecosystem, distilled from real incidents (mostly on the Argus project) rather than picked in the abstract. Each one exists because something concrete went wrong without it. Read this before customizing a template in [[Templates/Agents]] or [[Templates/Skills]] — it's the *why*, so you can judge edge cases the templates don't explicitly cover.

## Skill vs. Agent: a structural decision, not a style preference

> [!important] Dispatch capability decides the split
> If a component's job is to **orchestrate other agents** (claim work, dispatch specialists, act on their results), it must be a **Skill**, not an Agent. An Agent spawned as a subagent has no `Agent`/`Task` tool of its own — subagents cannot spawn subagents. A Skill runs inline in the main session and inherits its full tool access, including the ability to dispatch other agents.

This bit a real pipeline twice before the root cause was understood: a supervisor agent kept silently failing to run its own pipeline when launched via the Agent tool, because it had no dispatch capability. The first fix was a preflight check that told the caller to stop and re-invoke from the main session — a workaround. The durable fix was converting the supervisor from an Agent to a Skill entirely, which makes the failure mode structurally impossible instead of procedurally avoided.

**Rule of thumb:** if a component only *does* things (writes code, writes tests, audits, checks IaC) and returns a result, it's an Agent — isolate its context, since its scratch work (file reads, search results) doesn't need to pollute the caller. If a component *coordinates* things — including calling other agents — it's a Skill.

## Consent boundary: analysis is not authorization to act

> [!warning] Scoping questions get scoped answers
> If asked "what can run in parallel" or "which of these are ready," produce that analysis and stop. Do not treat it as approval to implement anything, even if the analysis makes the next step obvious. Only an explicit instruction to implement authorizes implementation.

This one is about trust calibration, not capability — the operator wants to delegate execution at high trust, but wants to be the one who decides *when* execution starts. Collapsing "here's what's possible" into "here's what I did" removes that checkpoint.

## Ground truth over expectation

> [!important] Verify against the current, actual state — not what should be true
> Before treating something as a bug, a missing dependency, or a blocker: check the current state of the code/branch/deployed artifact directly. A closed or merged dependency issue does not guarantee its full original scope shipped — issues get re-scoped across bail/re-scope cycles without every downstream reference being updated. A "bug" is sometimes just freshly-merged expected behavior nobody's session has caught up to yet.

Guessing from an issue's title, label, or from training-data memory of "how this usually works" is the failure mode this guards against. The fix is always the same shape: grep/read/query the actual current artifact before concluding.

> [!warning] Existence isn't sufficient — check that a condition is actually reachable
> A field or attribute existing in the schema is not the same as an acceptance criterion that depends on it being satisfiable. A real incident: a notification filter's acceptance criterion assumed a field would be set at evaluation time — the field existed, but it was only ever populated via a user action attached to the *same* notification the filter was supposed to gate, so it was always empty exactly when the filter needed to check it. Any non-trivial filter would have silently matched nothing, forever. Checking "does this field exist" would have missed this entirely; the check that catches it is "trace every write path that sets this field, and confirm it's actually set under the conditions this criterion assumes." See [[Templates/Skills/implement]] STEP 2c.

## Adversarial verification: assume both are wrong

> [!important] The last check in a pipeline must not grade the same homework twice
> If a producer (implementer) and a checker (test-writer) share the same blind spots — because the checker was scoped by the producer's own account of what it built — an independent audit stage is required, and it must derive its own expectations from intent *before* looking at the implementation or the existing test coverage. Reading the code first means only being able to imagine edge cases the code already has a branch for.

Concretely: derive an edge-case list from the stated guarantees first (Step 1), reconcile it against the existing spec/test coverage second (Step 2), and only then read the actual diff (Step 3). Discrepancies between the independently-derived list and the existing one are signal, not noise — don't silently merge them into one "combined" list; surface the mismatch (see [[Templates/Agents/edge-case-auditor]]).

> [!note] This isn't a theoretical risk specific to code review
> A large-scale, multi-institution red-teaming study of autonomous agents given real system access ("Agents of Chaos," arXiv 2602.20021, Feb 2026) documented agents reporting task completion while the underlying system state contradicted that report as one of eleven named failure modes — alongside cross-agent propagation of unsafe behavior between agents in the same system. Self-report is not a reliable substitute for independent verification for *any* agent in this pipeline, not just `core-implementer`/`test-writer` — a prompt-editing specialist's own account of "this edit is safe" needs the same independent check (see [[Templates/Agents/prompt-implementer]] and its independent auditor, [[Templates/Agents/prompt-auditor]]).

## Guardrails need an anchor outside the agent's own context, not just good prompting

> [!important] A rule written into an agent's prompt is not durable by itself
> Two real, separate findings converge on the same point. First, a documented incident: Meta's OpenClaw agent, given one explicit constraint ("don't act without my approval") while cleaning an inbox, had that exact constraint silently dropped by ordinary context-window compression — not an attack, routine memory management treating a 10-token safety instruction as low-priority filler among 50,000 tokens of conversation — and it went on to delete 200+ emails. Second, a governance-research finding: durable constraints on a self-modifying agent have to live at the institutional level (an audit trail, a review board) rather than the architectural level (a permission list, a whitelist, a code-based check), because a sufficiently capable agent can modify a secondary technical constraint the same way it modifies anything else it's asked to change — regulating an agent's reconfiguration with more agent-reachable configuration just relocates the same recursive problem ("Agent-initiated socio-technical reconfiguration," *Frontiers in Artificial Intelligence*, 2026).

> [!tip] This vault already does this in several places — this principle names why, it doesn't introduce anything new
> [[Templates/CI/README]] runs outside any agent's own session and can't be talked out of failing. [[Templates/Skills/spec]]'s two-step ADR confirmation doesn't take the spec skill's own account of "this drift is genuine" as sufficient — the Operator confirms twice, independent of what the agent believes happened. The adversarial-verification principle above dispatches its independent audit separately, never the same session grading its own work. None of these exist because someone anticipated context-compression or recursive self-modification specifically — they exist because a guardrail that lives only in one agent's own context, checked by that same agent, isn't actually a guardrail. Directly shaped [[Templates/Agents/prompt-implementer]]'s design.

## Every finding needs a disposition, not just a mention

> [!warning] A deferred finding that lives only in a chat message or a PR body is not tracked — it's forgotten
> Not every gap found by an audit has to be fixed before shipping. But every gap must be explicitly dispositioned: **fixed**, or **deferred with a durable anchor** — an inline `TODO(#issue)` at the exact code site, an ADR's "costs" section, or an `xfail` test with a `reason=` pointing at a tracking issue. A PR body is read once, by one reviewer, at merge time; nobody greps merged PR bodies six months later. The anchor is what makes the deferred item discoverable by the next person who touches that code, not just by someone who already knows to go looking.

## Incremental discipline scales trust

> [!tip] One change per stage, verified, then committed
> High-autonomy delegation (a supervisor running an entire implement → test → audit → PR cycle unsupervised) only stays trustworthy if each stage's output is checked before the next stage starts, and if staged/incremental work commits before moving on. Batching multiple fixes or multiple issues into one uncommitted pass removes the ability to tell which change caused a regression.

## Safety surfaces need a bright line, not a judgment call

> [!important] Some categories of change are never autonomous, regardless of how routine they look
> Any project adopting this pipeline should name its own "never touch autonomously" surface up front (financial calculations, auth/authz, secrets/IAM, or whatever the domain's equivalent is) and enforce it as a rule the agents are told about directly — not something inferred from context each time. Destructive or hard-to-reverse infrastructure actions (`apply`, deploys, secrets/KMS edits) require explicit human approval; read-only inspection and `plan`/`validate` do not.

> [!tip] The operational test underneath this: reversibility, and whether state/data/an external workflow actually changes — not a vague sense of risk
> "Does this action change state, move data, or trigger an external workflow that can't be safely undone" is a concrete, checkable bar, not a judgment call to make fresh each time — it's why `iac-specialist`'s plan-only/no-destructive-changes rule and a prompt-implementer's edit-tiering (see [[Templates/Agents/prompt-implementer]]) are the same underlying rule applied to different artifacts. This isn't unique to this vault's domain: ITIL's decades-old change-management practice tiers exactly this way — Standard changes (pre-approved, low-risk, routine) skip full case-by-case authorization; Normal changes get full assessment; Emergency changes bypass the normal process only for urgent risk mitigation. A safety surface is this vault's name for "always Normal-or-heavier, never Standard," not a bespoke invention.

## Ask an intent question only when evidence doesn't already resolve it

> [!important] Evidence-gathering must run before the question it's meant to ground, and an unambiguous answer means don't interrupt at all
> [[Principles#Safety surfaces need a bright line, not a judgment call]] and [[Principles#Consent boundary: analysis is not authorization to act]] establish *that* certain calls need an explicit human decision. This entry establishes *when* to actually stop and ask for one: only when the evidence gathered so far doesn't already resolve the question unambiguously. If it does — a docs directory already shows `[[wiki-link]]`/`> [!callout]` syntax, or an arc42-shaped `docs/architecture/` + `/adr/` pair already exists — auto-answer from that evidence and record the citation; don't interrupt to confirm something already settled by what's actually in the repo. Interrupting anyway isn't extra caution, it's noise that trains the reader to skim past every question this pipeline asks, including the ones that actually need their attention.

> [!tip] Existing precedent for the question shape itself
> [[Templates/Skills/spec]] STEP 3 already frames every Operator-judgment question the same way — stakes, a specific recommendation, the alternative and its cost, one question at a time, wait for the answer before continuing. Reuse that shape rather than inventing a new one; the discipline here is only about *whether* to ask, not *how*.

> [!note] Concrete incident this formalizes
> [[.claude/commands/project-lifecycle]]'s intent-gathering step got this ordering wrong in an earlier draft — it placed its safety-surface question ahead of its own evidence-gathering step, which would have asked from a generic list rather than the repo's actual candidate safety-surface items. Caught during design, not after another failed dry-run — the fix was reordering evidence-gathering to run first, then gating every question on "does this evidence already answer it," not just the safety-surface one specifically. A second, related lapse surfaced later, during audit rather than design: an earlier draft of the same command's STEP 3 correctly wrote this evidence-first *answer* for its docs-style question, but never actually wired that answer into STEP 3's own selection logic for the affected templates, so the old buggy behavior it was meant to replace would have kept firing regardless of the answer given. An evidence-gated answer that a downstream step never reads is functionally identical to never asking at all — recording the answer isn't enough; the step that would otherwise fall back to a weaker signal has to be the one that explicitly checks for it first.

## Capability grants are their own decision

> [!important] Widening what a component can do is a separate decision from the task that wants it
> "Sharpen this check" or "make this more thorough" is not automatic authorization to give a component new tool access — especially write/commit/push access it didn't have before. A real incident: asking to sharpen a spec-writing skill's dependency checks led to also drafting an ADR-writing capability for it, which required adding filesystem write and git commit/push to a skill that had previously only ever read code and edited an issue tracker. That capability expansion got caught and blocked by an automated permission gate — correctly — because the instruction that triggered the work ("sharpen the checks") didn't specifically authorize new git/write access, even though it was a reasonable thing to *want* while doing that work.

The fix is not to avoid capability changes — it's to surface them as their own explicit decision, separate from whatever task motivated them, and get a direct yes before proceeding. Bundling a permission-surface change into an unrelated edit (even a well-intentioned one) is exactly the kind of thing that should fail loudly rather than slip through. See [[Timeline]] for the incident this was drawn from.

## Fail loud, disclose by default

> [!tip] Silence is the failure mode, not the fallback
> "Tests pass" is wrong if some were skipped. "Feature works" is wrong if the specific edge case that was asked about wasn't verified. Default to surfacing what was carried silently — an assumption, a deferred risk, a workaround — rather than waiting to be asked. Severity decides *when* and *how loud*, never *whether*.

## Shared shapes need one definition, not one description per reader

> [!important] If a shape is assumed or referenced by more than one agent/skill, it needs to be written down once, in one place, and referenced — not re-described
> The trigger isn't "could this be a template" — nearly anything could. It's "does more than one file already point at this." A one-off shape that only ever appears in one place doesn't need extracting; that's premature abstraction, not de-duplication. But once a second file assumes the same issue format, comment format, or label, the two copies will drift the moment one of them is edited without the other being remembered.

This was found, not theorized: auditing this pipeline's own GitHub-interaction surface turned up two comment shapes (`implement`'s BAIL comment, the recurring-pattern comment `pipeline-review`/`ecosystem-diagnostician` both assume exists) and one issue-title convention (the ecosystem-fix tracking issue) that were each referenced by name from three separate files — and defined as literal text in *none* of them. Every file assumed one of the others had written it down. See [[Templates/GitHub/labels]], [[Templates/GitHub/issue-templates]], [[Templates/GitHub/comment-templates]] for the consolidated definitions, and [[Templates/Agents/triage]], [[Templates/Skills/spec]], [[Templates/Skills/implement]], [[Templates/Agents/ops-check]], [[Templates/Skills/pipeline-review]] for the callers now pointing at them instead of re-describing them.

> [!warning] This applies to structure, not to judgment
> A template should define the *fields* a shape needs, never script the *content* of a judgment-heavy field. `edge-case-auditor`'s finding descriptions and `ecosystem-diagnostician`'s "why this generalizes" reasoning stay free-form prose reacting to the specific instance — forcing those into fill-in-the-blank text would just produce well-formatted vagueness, the same failure [[Templates/Skills/spec]] already guards against by banning vague acceptance criteria. Templatize the envelope, not the reasoning inside it.

> [!tip] Ground new templates in established practice, don't invent from scratch
> Before drafting a new shared shape, check whether a real, citable convention already exists for it (a spec, a widely-used OSS pattern, a platform's own documented convention) and adapt that instead of designing from intuition. The GitHub layer above borrows its comment syntax from [Conventional Comments](https://conventionalcomments.org/) and its label namespacing and issue-template discipline from GitHub's own documented conventions — none of it was invented here. This is the same discipline [[Principles#Ground new additions in something citable|the general citation requirement]] asks for, applied specifically to template content.

## Ground new additions in something citable

> [!important] Every new Principle or Template should name what it's grounded in
> "My gut says X" isn't something a future reader can evaluate or push back on. Every addition should cite one of: an established engineering practice (SRE practices, the Twelve-Factor App, OWASP, a named framework), a documented agentic pattern (prompt chaining, orchestrator-workers, evaluator-optimizer — see Anthropic's "Building Effective Agents"), or a real incident this vault already has on record (see [[Timeline]]). "X because Y already established this, or incident Z proved it's needed" is checkable; a bare assertion isn't.

This isn't a bar for every trivial edit — it's for load-bearing additions specifically: a new Principle, a new Template, a new shared shape. Applied so far in this vault: the GitHub layer above (Conventional Comments, GitHub's own issue-form guidance), and the CI workflow in [[Templates/CI/README]] (scoped to exactly what `implement.md`/`pipeline-review.md` already assume, not the wider CI/CD suite the original idea guessed at).

> [!tip] A built-in's existence isn't the deciding factor — a concrete, nameable gap is
> A proposed `security-reviewer` agent turned out to duplicate Claude Code's own built-in `security-review` skill entirely — no project-specific gap existed, so it was redundant from day one. But the reverse is equally wrong: wrapping every built-in "as a base" out of caution, when there's nothing to add, is pure surface area for zero functional gain. Before designing a new specialist for a capability that sounds generic (security review, code review, dependency auditing), name the concrete gap the built-in structurally can't cover — a project-specific convention, label, or cross-file guarantee it has no way to know about — and if that gap is real, compose with the built-in (invoke it, then layer the project-specific check on its output) rather than replacing or discarding it. Claude Code built-ins are monolithic, callable only as a whole — this composition is always output-layering, never internals-layering.

## A sync mechanism needs both an automatic-narrow check and a periodic-broad sweep

> [!important] Neither half is optional — an automatic check alone misses aggregate patterns, and a periodic sweep alone misses real-time drift
> This pattern shows up three times in this ecosystem, independently arrived at each time: `vault-sync` (automatic, one PR at a time) is paired with `vault-auditor` (periodic, full-vault consistency sweep); the bail/spec-question recurrence check inline in `implement`/`spec` (automatic, exact-label match on one new event) is paired with `pipeline-review` (periodic, full-log sweep for patterns no exact-label match can see); and `ecosystem-sync` (automatic, one PR at a time, ports reusable `.claude/agents`/`.claude/commands` changes into this vault) exists for the same reason `vault-sync` does.

The concrete incident that motivates this for the vault-sync/ecosystem-sync case specifically: this vault went from "extracted from Argus" to visibly stale without anyone noticing — four agents (`ecosystem-diagnostician`, `ops-check`, `vault-auditor`, `vault-sync` itself) and two skills (`cleanup`, `pipeline-review`) existed in Argus's `.claude/` with no `Templates/` counterpart at all, discovered only when this gap was explicitly investigated rather than caught by anything running automatically. `ecosystem-sync` closes the automatic half of that gap going forward (see [[Timeline]]); this vault does not yet have a periodic-broad counterpart to it the way `vault-auditor` backs up `vault-sync` — see the open question below.

## Curated judgment and raw telemetry are two different logging problems, not one

> [!important] A hook can make a write mechanical; it cannot make the judgment a hook is writing
> [[Templates/Skills/pipeline-review]] and [[Templates/Agents/ecosystem-diagnostician]] consume a curated event log — `stage`, `blocker_type`, `category`, `recurrence_verdict` — and every one of those fields is a judgment call an agent makes, not something inferable from raw tool-call parameters. No hook alone can produce them. What a hook *can* fix is how that judgment reaches disk: today an agent must remember to compose and `Bash`-append a JSON line itself — self-report, discipline-dependent, and it costs output tokens to compose on every run. The fix is to redesign the write path, not to replace the judgment: the agent expresses the same judgment through a structured tool call (a dedicated tool/function with `stage`/`blocker_type`/`category` as typed parameters — the same decision it's already making in prose the moment it decides an event is log-worthy), and a `PostToolUse` hook keyed specifically to that tool's name performs the actual file write mechanically — guaranteed to fire every time that tool is invoked, no separate `Bash`-composition step, no chance of "forgot to log it." The consumer stays exactly what it is today; only the write path stops being self-reported.

> [!warning] Raw tool-call telemetry is a different stream, with a different — and still unbuilt — consumer
> A genuinely matcher-less `PostToolUse` hook (no tool-name restriction, firing on every `Read`/`Edit`/`Bash`/etc., for every agent) captures a different thing entirely: a full tool-call history with zero agent involvement and zero added tokens, the raw material for replay-based regression testing (re-running a past invocation's exact tool-call sequence against a new version of that agent's file and diffing the behavior). That stream's consumer is a future replay harness, not yet built — and its volume/storage characteristics are explicitly unsolved here: JSONL doesn't scale to that volume indefinitely, and a sampling or tiering strategy (likely graduating to SQLite/DuckDB at some point) is needed but not designed. Naming this as a deferred limitation, per [[Principles#Every finding needs a disposition, not just a mention]], not silently dropping it. Capturing "every decision" specifically — not just every tool call — is harder still, since a decision is often implicit in reasoning rather than a discrete call; see the open question below.

> [!tip] Grounded in a pattern this ecosystem already runs, not invented here
> Argus's own hooks already operate this way in production, outside any agent's context: a `PostToolUse` hook drives git auto-commit, and a `PreToolUse` hook enforces a worktree-violation guard. Both fire mechanically on every matching tool call regardless of what any agent remembers to do — the same property this principle asks the curated pipeline log's write path to have. See [[Principles#Guardrails need an anchor outside the agent's own context, not just good prompting]] for the general case this is one instance of.

> [!tip] The curated-log write path is now templated, not just described — one valid instantiation of the principle above, not a literal implementation of its specific mechanism
> [[Templates/Hooks/README]] ships a concrete design for the same underlying rule this section argues for (mechanize the write, never the judgment), but differs from the paragraph above in both hook type and where the write actually happens: instead of a `PostToolUse` hook keyed to a dedicated tool's name performing the write, it's a self-contained emission script (bash + `jq`, modeled on superflow's `sf-emit.sh` — typed `key=val`/`key:int=`/`key:bool=`/`key:json=` arguments, JSON built entirely through `jq --arg`/`--argjson` so it's injection-safe by construction, an allowlisted event-type check, threshold-based rotation) invoked via the generic `Bash` tool, paired with a `PreToolUse` guardrail hook that denies a direct write to the log file unless it routes through that script — the same shape as the already-shipped `worktree_violation_caught` hook cited above, generalized from one mechanically-written event to the full curated vocabulary. The guardrail is scoped to block the bypass, not ad hoc logging itself: a free-form session logging `note`/`incident`/`decision` by hand still goes through the same one-line script call, never a blocked path. Both the script and the hook fail loud on any validation failure, per [[Principles#Fail loud, disclose by default]] — unlike `sf-emit.sh`'s own fail-open default. As with every template in this vault, this documents and exemplifies the pattern; wiring it into any specific adopting project's live `.claude/settings.json` is separate work in that project's repo.

## Comment discipline for code-writing specialists

> [!important] Default to no comment; write one only when the WHY is non-obvious
> `core-implementer`, `test-writer`, and `iac-specialist` had no general comment-writing convention — each carries only narrow, mechanical rules (safety-surface inline comments and public-function doc-comments on `core-implementer`), with no stated default and no guidance on avoiding noise; `test-writer` and `iac-specialist` said nothing about comments at all, despite IaC being exactly where "why this weird resource config exists" most needs capturing. The convention: default to no comment. Write one only when the WHY is genuinely non-obvious — a hidden constraint, a subtle invariant, a workaround for a specific bug, something a future reader would be surprised by. Never explain WHAT the code does — that belongs in code written clearly enough not to need narrating. Never reference the current task, fix, or caller by name — that rots the moment the codebase moves past this change, and belongs in the commit message instead.

> [!tip] Not invented here — the documented industry convention, arrived at independently twice
> Google's own style guides and *Software Engineering at Google* (O'Reilly; Winters/Manshreck/Wright), ch. 8 ["Style Guides and Rules"](https://abseil.io/resources/swe-book/html/ch08.html): implementation comments exist to "justify or highlight non-obvious choices, explain tricky bits, and underscore important parts" — they explain intent, not restate the obvious. Robert C. Martin, *Clean Code* (2008), ch. 4 "Comments": comments are a fallback for failing to express intent in code; noise comments (`i++; // increment i`) and journal/attribution/history comments are called out specifically as harmful, because source control already tracks that history — the same reasoning behind "never reference the current task/fix/caller, put that in the commit message instead." [CUSTOMIZE] If an adopting project's own CLAUDE.md already states a different comment convention, that convention wins — this is a sensible default, not an override of a project's explicit choice.

## Templates carry a real semver, not just a version-string decoration

> [!important] major.minor.patch, defined by contract impact on an adopter's already-deployed copy — not by how much text changed
> A template's version tracks the same thing a versioned API's does, per the same discipline [semver.org](https://semver.org/) formalizes for general software: whether an adopter who already pulled a copy into their own `.claude/agents`/`.claude/commands` and is running it today would need to change anything to keep using the new version.
>
> - **Major (breaking)** — anything that could break an adopter's already-deployed copy if they pulled the update without reading it first: the return-contract changes shape (e.g. an Agent's DONE/BAIL/NEEDS-CONFIRMATION structure, or a field inside it, added/removed/renamed); an existing `[CUSTOMIZE]` placeholder an adopter has already filled in is renamed or removed; a `tools:`/`allowed-tools:` grant is narrowed (something the adopter's copy relies on being available is taken away); a Skill's `$ARGUMENTS` contract changes shape (a positional argument reordered, removed, or given new required structure).
> - **Minor (non-breaking, additive)** — a backward-compatible addition: a new optional step or check that doesn't change existing output shape; a new `[CUSTOMIZE]` placeholder that ships with a safe default so an adopter who does nothing still works; widening a `tools:`/`allowed-tools:` grant (adding capability never removes any that was already there).
> - **Patch** — wording, clarity, typo fixes, added cross-references or grounding citations — no behavior change at all.
>
> When in doubt between two tiers, pick the higher one — same discipline as [[Templates/Agents/prompt-implementer]]'s own edit-tiering, and for the same reason: the cost of an unnecessarily-loud version bump is small, the cost of an adopter silently landing on a breaking change labeled non-breaking is not.

> [!note] Sanity-checked against this vault's own history, not designed in the abstract
> Before this rule shipped it was checked against real past edits to `Templates/Agents/*.md`/`Templates/Skills/*.md`: widening `prompt-implementer`'s own description to cover this vault's `.claude/commands/` (`f3c036a`) is additive scope with nothing existing removed — **minor**. Adding the comment-discipline convention to `core-implementer`/`test-writer`/`iac-specialist` (`0e03859`) is a new guidance line, no existing behavior removed — **minor**. `31bcc1b` (`triage.md`/`spec.md`) actually spans two tiers within the same commit: replacing the inline literal triage-brief/full-spec format with a cross-reference to the single canonical definition in [[Templates/GitHub/issue-templates]] produced the exact same output shape through a different path — **patch** — but the same commit's `spec.md` diff also adds a new required field to the ADR-drafting instruction ("Draft a new ADR... with one addition before Decision: **Alternatives considered**...") — a backward-compatible content addition to what the skill produces — **minor**. Per the rule above, a commit spanning two tiers classifies at the higher one overall, so `31bcc1b` as a whole is **minor**, not patch. Rewording `pipeline-review.md`'s note about *how* a log gets written, while stating explicitly that "the schema this skill reads is unaffected either way" (`359bb2d`) — **patch**. This vault's own history doesn't yet contain a genuine breaking change to an already-shipped template — the major case is grounded in the definition above, not yet in a real instance of this vault's own history; treat the first real one as worth double-checking against this rule rather than assuming it's obviously covered.

> [!warning] Skills don't have as crisp a return-contract as Agents — classify by the anchors that do exist
> An Agent's DONE/BAIL/NEEDS-CONFIRMATION-style output is an explicit, textual contract; most Skills don't have an equivalent single return shape (a few do, e.g. [[Templates/Agents/prompt-implementer]]'s own DONE/NEEDS-CONFIRMATION/BAIL). For a Skill, anchor the major/minor call on what's actually contractual: its `$ARGUMENTS` shape, its `allowed-tools:` grant, and any `[CUSTOMIZE]` placeholder an adopter fills in — the same three anchors the rule above already names. A Skill edit that touches none of those, and doesn't reorder or remove an existing required step a caller depends on, is minor at worst.

## Template distribution must be relevance-scored, never install-everything

> [!important] A project needing zero templates from a category is a correct outcome, not a gap
> Any mechanism that installs or proposes this vault's agent/skill templates into a target project must evidence-score which specific templates that project actually needs — never a fixed default set. No `iac-specialist` without detected IaC files, no `ops-check` without a detected production log source. A docs-only project can correctly end up with zero installed pipeline templates.
>
> This is what "universal" is supposed to mean for this ecosystem — not that every project gets every agent, but that the same template set can correctly resolve to a different subset for every project.

The concrete incident: the original `/onboard` skill installed the full template set into every project it touched, regardless of fit. Per the Operator directly: "not every project needs every agent template we have... it should decide what agents and skills to give it based on what the project is doing." This was one of two named reasons `/onboard` was torn down same-day it was built (the other — built without being asked — is a process failure, not a design one; see [[Principles#Consent boundary: analysis is not authorization to act]]).

## Claude Code's own plugin system already solves push-updates for skills — but not, right now, for this vault's agent customization

> [!important] The mechanism isn't a hypothesis — it's documented, and this session's own installed skills are live proof
> Investigated for issue #37, ahead of building any push-update infrastructure of this vault's own. Plugin **skills/commands** are genuinely push-update-solved by Claude Code itself: a plugin's skills get copied into a local cache (`~/.claude/plugins/cache`) and re-synced on update — not live reference, but functionally equivalent to "consumers get updates without a manual push mechanism." This session's own installed plugin-namespaced skills (`aws-serverless:*`, `context7:*`, `claude-md-management:*`) were never copy-pasted into this vault; they just work, on exactly this mechanism.

> [!note] Agents also support genuine per-install customization, via `userConfig`
> A plugin's `userConfig` schema (declared in `plugin.json`) drives an interactive configuration dialog Claude Code prompts the consumer with, and non-sensitive answers substitute into an agent's markdown body via `${user_config.KEY}` placeholders (and into a bundled MCP server's environment variables). Fields can be marked `required: true` (validation fails if left blank) or optional with a default. Source: Claude Code's own plugins-reference documentation.

> [!warning] Real mechanism, wrong shape for this vault's actual customization surface, right now
> `userConfig` values are constrained to `string`, `number`, `boolean`, `directory`, `file` — short scalar fields, not the multi-paragraph prose this vault's `[CUSTOMIZE: ...]` placeholders actually are (e.g. "describe your domain, e.g. these agents operate in a fintech company handling PCI-scope review, escalate to #security-eng"). Fitting the `userConfig` model would mean decomposing that prose into several named short fields spliced into fixed surrounding text the plugin author commits to in advance — a real rewrite of every `Templates/Agents/*.md`/`Templates/Skills/*.md` file's customization surface, not a drop-in swap of distribution mechanism. Distribution also requires infrastructure this vault doesn't have: a `marketplace.json` plus a `plugin.json` per template — raw git-repo file references aren't installable on their own, per Claude Code's own plugin-marketplaces documentation. Two UX gaps remain undocumented on top of that: whether the initial `userConfig` prompt fires at install time or requires a session restart/explicit trigger, and whether there's any flow to reconfigure a value after initial install (best evidence: no — a consumer would hand-edit `~/.claude/settings.json` directly, under `pluginConfigs[<plugin-id>].options`).

Weighed against this vault's current adopter count — one, Argus, per [[Adopters]]/`adopters.yaml`, with #63 about to add a second — building this now is the same speculative-ahead-of-need mistake [[Principles#Template distribution must be relevance-scored, never install-everything]] already warns against, just applied to a distribution mechanism instead of a template set; issue #7 (push-updates) is already gated behind "2+ real adopters" for the same reason. This is not a clean kill, though: the mechanism genuinely exists — the pre-shaping hypothesis that it might not was wrong — it's evaluated and correctly deferred, not dismissed. Revisit trigger: adopter count growing past what manual copy-updates can handle, or `userConfig` gaining richer/multiline text support. Full investigation and pre-mortem: issue #37.

## Open questions

> [!question] What's the storage/tiering strategy for raw tool-call telemetry, and how does "every decision" (not just every tool call) get captured?
> Deferred out of scope by the capture-layer split above: a matcher-less `PostToolUse` hook can capture every tool call cheaply, but JSONL doesn't scale to that volume indefinitely, and no sampling/tiering strategy is designed yet. Separately, "every decision" is a harder capture target than "every tool call" — a decision is often implicit in reasoning, not a discrete call. Open sub-question: capture reasoning/thinking output where the runtime exposes it, or keep pushing decision points into structured tool calls the way the curated pipeline log now does. Distinct from the cross-project aggregation question below — that one assumes a populated log already exists; this one is about what populates the raw telemetry stream in the first place, and at what cost.

> [!question] Should this vault have a periodic-broad counterpart to `ecosystem-sync`, the way `vault-auditor` backs up `vault-sync`?
> `ecosystem-sync` only catches drift one PR at a time, in whichever project has it wired in. It can't catch: a project that adopted the templates by hand and never wired `ecosystem-sync` at all, a change made directly to a `Templates/` file in this vault that drifts from what an adopting project's live copy actually needs, or an aggregate pattern across multiple adopting projects' `.claude/` history that no single PR's diff would surface. A `vault-auditor`-shaped agent for this vault — reading every adopting project's current `.claude/agents`/`.claude/commands` against `Templates/`, flagging drift either direction — would need a registry of which projects have adopted this pipeline at all. That registry now exists ([[Adopters]], `adopters.yaml`), seeded so far with Argus alone; worth building the auditor once there are two or three adopting projects registered to actually audit against, not before.

> [!question] Should this vault aggregate cross-project pipeline telemetry, and if so, when?
> Argus's `PIPELINE_LOG.jsonl` already detects recurrence *within* one project — the same `stage`+`blocker_type`, or the same `spec_question` category, showing up twice triggers `ecosystem-diagnostician` to judge whether it's a genuine repeat before drafting a fix. The open question is whether a generalized version of that log belongs in this vault, so the same recurrence check can run *across* projects, once more than one project is emitting this data.
>
> **Update:** `ecosystem-diagnostician` is now ported to [[Templates/Agents/ecosystem-diagnostician]] — that blocker is resolved. What's still missing is the telemetry stream itself: no project other than Argus emits a pipeline log yet, so there is nothing to aggregate or cross-reference. `ecosystem-sync` (see [[Templates/Agents/ecosystem-sync]]) now gives this vault a live mechanism for "any project pushes into this vault," the same shape `Insights/`/`Growth/` already established — but it carries `.claude/agents`/`.claude/commands` content, not pipeline-log telemetry. Extending it (or adding a sibling mechanism) to also carry a generalized log event is a plausible next step, not yet done.
>
> **For:** the mechanism to receive cross-project pushes now exists in more than one form (`Insights/`/`Growth/`, `ecosystem-sync`) — adding a telemetry stream would reuse an established pattern, not invent a new one.
>
> **Against / unresolved:** (1) Still only one project (Argus) would emit this data today — building the aggregation now is seeding future capability, not solving a live problem, until a second adopting project exists. (2) Argus's schema (`blocker_type` values like `kms-secrets-touch`, specialist names, `safe_surface`) is project-specific and shouldn't be copied verbatim — a generalized event would need to shrink to project name, stage, a normalized category label, and `recurrence_verdict`, not the full Argus field set. (3) Whether a repeated label match should ever *auto*-promote a pattern into this file, or whether that judgment must stay manual as it is today — Argus's own `recurrence_verdict` field already distinguishes "recurring" from "same label, different actual cause," which is a judgment call a label match alone can't make.
>
> Still open: the generalized event schema, whether to start capturing now purely to backfill history before a second project exists, and whether the cross-project reader of this log should be `ecosystem-diagnostician` itself (already generalized) or a new template entirely.
